# X4Native — SDK Contract & Risk Assessment

## Internal C ABI

Extensions interact through `x4native.h` (the C++ namespace API), which wraps a flat C ABI that crosses DLL boundaries safely. Extension developers never use this layer directly.

### Required Exports

Generated by `X4N_EXTENSION` / `X4N_SHUTDOWN` macros:

```cpp
extern "C" __declspec(dllexport) int x4native_init(X4NativeAPI* api);
extern "C" __declspec(dllexport) void x4native_shutdown(void);
extern "C" __declspec(dllexport) int x4native_api_version(void);
```

### Internal Types

```cpp
typedef void (*X4NativeEventCallback)(const char* event_name,
                                      void* data, void* userdata);

typedef struct X4HookContext {
    const char* function_name;
    void**      args;           // Array of pointers to each argument
    void*       result;         // Pointer to return value buffer
    int         skip_original;  // OR-gate: any setter causes skip
    void*       userdata;
} X4HookContext;

typedef int (*X4HookCallback)(X4HookContext* ctx);
```

### X4NativeAPI Struct

```cpp
struct X4NativeAPI {
    int api_version;

    // Event System
    int  (*subscribe)(const char* event_name,
                      X4NativeEventCallback callback, void* userdata);
    void (*unsubscribe)(int subscription_id);
    void (*raise_event)(const char* event_name, void* data);

    // Logging (routes to global x4native.log; per-extension routing uses _reserved[4])
    void (*log)(int level, const char* message);

    // Info
    const char* (*get_game_version)(void);
    const char* (*get_x4native_version)(void);
    const char* extension_path;

    // Game API
    struct X4GameFunctions* game;
    void* (*get_game_function)(const char* name);
    int game_func_count;
    int game_types_build;

    // Lua Bridge
    int  (*raise_lua_event)(const char* event_name, const char* param);

    // Hook System (wrapped by x4n::hook)
    int  (*hook_before)(const char* function_name,
                        X4HookCallback callback, void* userdata);
    int  (*hook_after)(const char* function_name,
                       X4HookCallback callback, void* userdata);
    void (*unhook)(int hook_id);

    // Stash — in-memory key-value, survives /reloadui + hot-reload, lost on game exit
    // Wrapped by x4n::stash in the C++ SDK.
    int         (*stash_set)(const char* ns, const char* key,
                             const void* data, uint32_t size);
    const void* (*stash_get)(const char* ns, const char* key,
                             uint32_t* out_size);
    int         (*stash_remove)(const char* ns, const char* key);
    void        (*stash_clear)(const char* ns);

    void* _reserved[24];  // ABI-compatible expansion
};
```

The `_reserved` slots allow adding new function pointers without breaking existing extensions.

## Risk Assessment

### Risk Matrix

| | Low Impact | Medium Impact | High Impact |
|---|---|---|---|
| **High Likelihood** | | Anti-cheat false positive (online features disabled) | Game crash from bad hooks (version mismatch) |
| **Medium Likelihood** | Log file grows large | Extension crash propagation | Heap corruption from rogue extension |
| **Low Likelihood** | Config parsing errors | LuaJIT state corruption | Save corruption |

### Key Mitigations

| Risk | Mitigation |
|---|---|
| Game update breaks hooks | Version detection + offset database; refuse to hook unknown versions |
| Extension crashes game | SEH wrapping + auto-disable of crashing extensions |
| LuaJIT state corruption | Minimize direct `lua_State` access; prefer event-based communication |
| Online features disabled | Already happens with Protected UI Mode off; documented clearly |
| Save corruption | Never write to save data from native code; use Lua/MD bridge or stash for session data |
| Malicious extensions | Clear documentation that extensions run with full process access |

## Comparison with sn_mod_support_apis

| Aspect | sn_mod_support_apis | X4Native |
|---|---|---|
| Architecture | Lua + thin C DLL (pipes) | Lua bootstrap + C++ framework |
| DLL Purpose | Named pipes (IPC) | Multi-purpose hosting framework |
| Extension Loading | N/A (monolithic) | Dynamic: scans and loads extension DLLs |
| Communication | Named pipes to external process | In-process: direct function calls |
| Function Hooking | None | MinHook inline hooking |
| Event System | Lua `RegisterEvent` | Native C++ pub/sub + Lua bridge |
| Performance | Pipe latency + 1-frame MD delay | Near-zero overhead (direct calls) |
| Crash Safety | Python server is out-of-process | In-process (SEH wrapping) |
| Version Stability | High (public Lua API only) | Medium (hooks depend on game version) |

These approaches are **complementary**: `sn_mod_support_apis` for external tool integration, X4Native for in-process game modification.
